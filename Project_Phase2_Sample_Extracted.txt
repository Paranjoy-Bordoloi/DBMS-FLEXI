Title of the project:
Personal Finance Management System
About the project:

The Personal Finance Management System (PFMS) is a sophisticated application designed to
empower users in managing their finances with precision and ease. Developed using Java Swing for
the user interface and MySQL for robust database management, PFMS offers a comprehensive suite
of features tailored to financial tracking and analysis. Through PFMS, users can register and create
multiple accounts, each meticulously categorized to represent different financial aspects such as
savings, checking, and investments. Income and expenses can be recorded seamlessly, with detailed
information including dates, sources, categories, and amounts, ensuring a granular understanding of
financial activities.
Furthermore, PFMS facilitates effective budgeting and goal setting by allowing users to establish
budgets for various expense categories and define target amounts to achieve specific financial
milestones. The system's architecture prioritizes data integrity and security, employing foreign key
constraints and triggers to maintain consistency within the database while ensuring that only
authorized users can access and modify financial data. Additionally, PFMS offers visualization tools
such as charts to analyze monthly financial data, empowering users to make informed decisions and
gain insights into their financial habits and trends. With its scalable and flexible design, PFMS stands
as a reliable companion for individuals seeking to take control of their financial well-being.

Addressed paint points:
The pain point addressed by the Personal Finance Management System (PFMS) lies in the complexity and
disorganization inherent in manual financial tracking. Many individuals struggle to manage multiple accounts,
track various income sources, and categorize expenses efficiently. PFMS offers a solution by providing a
user-friendly interface and robust database management, streamlining the process of recording and organizing
financial data. By centralizing income, expenses, and budgets within a structured framework, PFMS eliminates
the frustration of scattered financial records and enables users to gain clarity and control over their finances
effectively.

Anomalies:
A database anomaly is a fault within a database, which can occur because of poor planning or
when everything is stored in a flat database. These anomalies naturally occur and result in
data that does not match the real-world the database purports to represent.
Anomalies occurring in our database:
1) 1NF: Insert Anomaly: In the Budget table, having multiple category names within a single cell
(multi-valued attribute CategoryName) violates the first normal form (1NF) > BudgetID=3. This means if
we want to insert a new budget entry with multiple categories, we need to repeat all other attribute
values for each category, leading to redundant data.
Update Anomaly: Updating a category name for a specific BudgetID would require updating multiple
rows if the BudgetID has multiple categories associated with it, leading to inconsistencies.
Delete Anomaly: Deleting a specific category associated with a BudgetID would lead to loss of
information about other categories if they are stored within the same cell, potentially breaking the
budget entry.

2) 2NF: Insert Anomaly: To insert a new user, we must provide all attributes including UserID, which may
not always be available or relevant at the time of insertion.
Update Anomaly: Updating information about a user requires updating potentially multiple rows if the
user has multiple accounts, leading to inconsistency.
Delete Anomaly: Deleting a user might result in losing information about the associated accounts,
leading to data inconsistency.

Normalization:
A normalization procedure, which combines and splits tables, is usually sufficient to remove
this. By normalizing the database, we reduce the likelihood of creating tables that
generate anomalies. Normalization is necessary if you do not do it then the overall integrity
of the data stored in the database will eventually degrade. Specifically, this is due to data
anomalies.
Types of normalization
1. Unnormalized form or UNF:
It is the simplest database model also known as non-first normal form (NF2).A UNF
model will suffer problems like data redundancy thus it lacks the efficiency of
database normalization.
2. First Normal Form or 1NF:
A relation is in first normal form only if the relational table doesn’t contain any
multivalued attribute but contains only single-valued attributes.
3. Second Normal Form or 2NF:
A relation is in second normal form if:

• It is in first normal form or 1NF
• It doesn’t contain any partial dependencies. (It shouldn’t have any non-prime
attribute which is functionally dependent on any proper subset of the candidate key of
the relation.).
4. Third Normal Form or 3NF:
Let R be the relational schema, X->Y any non-trivial functional dependency over R is
in 3NF if:
• R should be in 2NF
• X should be candidate key or superkey, or
• Y should be prime attribute

Normalization occurring in our code:
1) 1NF: In the unnormalized version of the Budget table, the presence of the multi-valued attribute
CategoryName violates the first normal form (1NF) by allowing multiple values within a single cell. To
adhere to 1NF, we create a separate table, Budget_Category, to store individual category names. This
normalization ensures each attribute in the Budget table holds atomic values, promoting data integrity
and efficient querying.

2) 2NF: The attributes Fname, Lname, Username, and Email are functionally dependent only on UserID,
but not on the entire primary key (which includes AccountID). They describe the user associated with
the account, not the account itself. This is a partial dependency because these attributes depend on
part of the primary key.

To remove partial dependency and achieve 2NF, we need to separate user-related attributes into a
separate table.

3) 3NF: The attributes CategoryName and CategoryDescription are functionally dependent on
CategoryID, but they are indirectly dependent on the primary key through RecordID. These attributes
describe the category associated with the expense, which is determined by CategoryID, not RecordID.
Therefore, it's a transitive dependency because it depends indirectly on the primary key through
another non-key attribute.

To remove the transitive dependency and achieve 3NF, we need to separate the attributes related to
categories into a separate table.

returns decimal(10,2)
reads sql data
begin
declare cur_bal decimal(10,2);
select sum(i.amount) - sum(e.amount) into cur_bal
from account a
join records r on a.accountid = r.accountid
join income i on r.recordid = i.recordid
join expenses e on r.recordid = e.recordid
join user u on u.userid = a.userid
where u.username = in_username;
